Implementation Approaches

Choose how we apply the HUD-style changes

A: CSS Variables Only (Recommended)

Strategy: Change --radius to 0rem in globals.css, update CSS custom properties

Pros

  • Fastest implementation (~1 file change)
  • Consistent radius removal via Tailwind's var(--radius-*)
  • Easy to revert or adjust later
  • No component file edits needed

Cons

  • Some hardcoded rounded-* classes won't change
  • Less control over individual components

B: Component-Level Updates

Strategy: Edit each shadcn component file to remove rounded-* classes, add angular corners

Pros

  • Complete control over each component
  • Can add corner brackets to cards
  • No hardcoded radius left behind

Cons

  • Edit 15-20 component files
  • More maintenance if shadcn updates components
  • Slower implementation

C: Hybrid Approach

Strategy: CSS variable change + targeted component edits for cards (corner brackets) and buttons

Pros

  • Best of both worlds
  • Corner brackets on cards for HUD feel
  • Fast radius removal via CSS vars
  • Minimal component edits (3-5 files)

Cons

  • Slightly more complex
  • Two different change patterns

My Recommendation: Hybrid Approach (C)

Since you want "subtle professional" with "minimal orange accents" and "all components" radius removed:

  • Set --radius: 0rem in globals.css (removes all Tailwind radius-* classes)
  • Add orange as a custom CSS property for minimal accents
  • Edit card.tsx to add subtle corner brackets (L-shaped borders in orange)
  • Update button variants to use sharp corners
  • Update dashboard page to use uppercase labels and monospace where appropriate
